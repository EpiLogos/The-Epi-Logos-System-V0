## Story
**As an** Admin user,
**I want** the system to automatically preprocess uploaded documents using Docling and ingest them into LightRAG,
**so that** documents are quickly queryable through the Gnostic namespace with coordinate-aware context while preserving raw chunks for future deep analysis.

## Dependencies
- **Story 02.15**: LightRAG Foundation Integration (REQUIRED) - Provides the foundational LightRAG service infrastructure for document ingestion and hybrid vector-graph operations
- **Story 08.02**: Epii Knowledge Base Management (REQUIRED) - Provides admin interface for document uploads that trigger this pipeline

## Acceptance Criteria
1. The system must integrate Docling as the preprocessing layer for all uploaded Bimba documents (PDF, DOCX, PPTX, XLSX, HTML supported).
2. Docling must extract structured chunks preserving hierarchical context (headings, sections, tables, reading order).
3. The pipeline must generate a lightweight document-level summary (single LLM call) to provide contextual grounding for chunk embeddings.
4. Each chunk must be wrapped with contextual metadata: document summary, Bimba coordinate, subsystem info, and structural position from Docling.
5. Contextualized chunks must be ingested into the existing LightRAG service (Neo4j + Qdrant) for immediate queryability.
6. Raw Docling chunks must be persisted separately to enable future re-processing without re-parsing (supports Story 08.09 analysis pipeline).
7. The pipeline must be triggered automatically when admins upload documents via Story 08.02 interface.
8. The system must provide pipeline status tracking and notifications for admin users.

**Docling API Patterns**: [Source: https://docling-project.github.io/docling/]
```python
from docling.document_converter import DocumentConverter

# Basic usage pattern
converter = DocumentConverter()
result = converter.convert(source_path_or_url)  # Returns ConversionResult

# Access converted document
doc = result.document  # DoclingDocument object
doc_dict = doc.export_to_dict()  # Structured JSON with chunks, metadata, hierarchy

# ConversionResult attributes
result.status      # PENDING, SUCCESS, PARTIAL_SUCCESS, FAILURE
result.pages       # List of converted pages
result.errors      # List of errors during conversion
result.timings     # Performance metrics
```

**Critical Implementation Notes**:
- Docling parsing is **SYNCHRONOUS** (library limitation - no async support)
- Use `export_to_dict()` to get structured chunks with hierarchical metadata
- Each chunk preserves: content, heading_level, section_title, reading_order, table_index
- Local execution (~2-5 seconds per document), no external API calls

**Async/Sync Pattern Strategy**: [CRITICAL for correct implementation]
```
Component                    | Pattern | Rationale
----------------------------|---------|------------------------------------------
Docling preprocessing       | SYNC    | Library limitation (no async support)
Document summarization      | ASYNC   | LLM call via Gemini async API
Context wrapping            | SYNC    | Pure string manipulation (<1ms)
LightRAG ingestion          | ASYNC   | Database writes (Neo4j + Qdrant)
MongoDB chunk storage       | ASYNC   | Database writes via motor/async pymongo
Pipeline orchestrator       | ASYNC   | Coordinator calling sync/async components
Redis status tracking       | SYNC    | Fast operations, no blocking concerns
```

**Implementation Pattern**:
```python
async def pipeline_orchestrator(doc_upload):
    # 1. Sync Docling preprocessing (run in thread pool)
    chunks = await asyncio.to_thread(docling_preprocessor.parse, doc_upload.file)

    # 2. Async summary generation
    summary = await summary_service.generate_summary(doc_upload.content)

    # 3. Sync context wrapping
    wrapped_chunks = chunk_wrapper.wrap_with_context(chunks, summary, doc_upload.coordinate)

    # 4. Async LightRAG ingestion
    await lightrag_service.ingest_chunks(wrapped_chunks)

    # 5. Async MongoDB raw chunk storage
    await chunk_store.save_raw_chunks(chunks, doc_upload.id)

    # 6. Sync Redis status update
    redis_client.set(f"pipeline:{doc_upload.id}:status", "complete")
```

**Pipeline Flow**: [Source: Dual-pipeline architecture design]
1. **Upload** (Story 08.02): Admin uploads document with coordinate via frontend
2. **Trigger**: Backend triggers async pipeline orchestrator
3. **Docling Parse**: SYNC extraction of structured chunks (run in thread pool)
4. **Summarize**: ASYNC single Gemini LLM call generates document-level summary
5. **Wrap Chunks**: SYNC context wrapper (summary + coordinate + subsystem + position)
6. **Ingest to LightRAG**: ASYNC store in Neo4j (graph) + Qdrant (vectors)
7. **Store Raw Chunks**: ASYNC persist to MongoDB `doc_chunks` collection
8. **Notify Admin**: Update Redis pipeline status to "complete"
