# Story 08.13: MEF Resonance Analysis via Parashakti Agent

## Status
Approved

## Story
**As a** Etymology Atelier user exploring word communities,
**I want** MEF (Meta-Epistemic Framework) resonance analysis to discover related Bimba coordinates through 6-lens cognitive framework,
**so that** I can understand deeper archetypal, causal, logical, processual, epistemic, and divine-scalar connections within my etymological research.

## Acceptance Criteria

1. **Bimba Node Configuration (#5-4.2 Parashakti)**
   - #5-4.2 Parashakti node has complete `f_capability_mef_analysis_*` properties (description, lenses, lens_0 through lens_5, cognitive_approach)
   - #5-4.2 has `f_workflow_mefAnalysis_*` properties with 4 stages (Context Loading, 6-Lens MEF Analysis, Bimba Resonance Discovery, Resonance Synthesis)
   - Prakasa can load MEF capability and workflow from #5-4.2
   - Properties are structured as specified in implementation plan (MEF comprehensive synthesis document)

2. **Parashakti Agent Tool Registration**
   - Parashakti agent creates with `deepseek:deepseek-chat` model (64K context, reasoning-capable)
   - Agent has 3 MEF tools registered via hardcoded pattern in `constellation.py` (following Epii EA pattern)
   - Tools: `semantic_coordinate_discovery`, `get_direct_children`, `get_node_relationships`
   - No `f_tools` property in Neo4j (tools hardcoded, not stored in graph)

3. **Backend MEF Service**
   - `backend/epi_logos_system/cag/graphiti/mef_service.py` created with `run_mef_analysis()` function
   - Service loads community context, invokes Parashakti agent, parses MEF results
   - `store_bimba_resonance()` function creates BimbaResonance nodes with EA+Episodic labels
   - `clear_existing_resonances()` function supports re-analysis workflow
   - BimbaResonance nodes have `etymology_session_id` property for session queries
   - DeepSeek reasoning chain stored in `deepseek_reasoning_chain` property

4. **API Endpoints**
   - `POST /api/graphiti/etymology/communities/{community_id}/analyze-mef` triggers async MEF analysis (manual trigger)
   - Endpoint validates user owns community before queueing background task
   - Returns success with `is_reanalysis` flag if community had prior resonances
   - `GET /api/graphiti/etymology/communities/{community_id}/resonances` returns BimbaResonance array
   - Resonances include coordinate details, MEF reasoning summary, DeepSeek reasoning chain, lens-specific insights

5. **Neo4j Schema & Storage**
   - BimbaResonance node constraints: `uuid` unique, indexes on `resonance_strength`, `resonance_type`, `etymology_session_id`
   - Schema: `(community:EA:Community)-[:RESONATES_WITH]->(resonance:BimbaResonance:EA:Episodic)-[:TARGETS]->(coord:BimbaNode)`
   - CRITICAL: BimbaResonance nodes have BOTH `EA` and `Episodic` labels
   - Properties: `resonance_type`, `resonance_strength` (0.0-1.0), `description`, `detected_via_lens`, `detected_via_tool`, `reasoning_summary`, `deepseek_reasoning_chain`, `mef_archetypal`, `mef_causal`, `mef_logical`, `mef_processual`, `mef_meta_epistemic`, `mef_divine_scalar`

6. **Frontend Integration**
   - TypeScript types updated in `frontend/src/types/etymology.types.ts` with `BimbaResonance` interface
   - `EtymologyCommunity` interface includes `bimba_resonances` array
   - "Analyze MEF" button in Bimba Resonances section (manual trigger)
   - Resonances display with coordinate name, strength, type, description
   - MEF reasoning summary visible in UI
   - DeepSeek reasoning chain expandable in UI
   - MEF lens insights expandable by lens (6 lenses)

7. **Quality Requirements**
   - MEF analysis completes in <60 seconds for typical community
   - Resonance strength calculations meaningful (not random), based on MEF alignment + semantic similarity
   - Resonance descriptions reference specific lens insights
   - Tool usage balanced across semantic/structural/canonical discovery modes
   - Reasoning summary accessible to non-technical users
   - No duplicate resonances for same coordinate
   - Error handling for failed MEF analysis doesn't break community creation

8. **Architectural Compliance**
   - Follows Prakasa protocol (capability + workflow pattern)
   - Tool whitelist follows Epii EA hardcoded pattern in `constellation.py` (NOT Neo4j f_tools)
   - Maintains trilaminar boundaries (Frontend ↔ Backend ↔ Agentic)
   - Async execution doesn't block orchestrator
   - Frontend types match backend response structure

- 6-lens cognitive framework: Archetypal, Causal, Logical, Processual, Meta-Epistemic, Divine-Scalar

### Important Implementation Details

**Tool Registration Pattern** (CRITICAL):
- Follow Epii EA pattern: tools hardcoded in `constellation.py`, NOT stored in Neo4j
- Current `f_tools` property in Neo4j is NOT used for actual filtering (shared_tools.py line 1303 falls back to setup_all_cag_tools)
- Register tools AFTER `setup_selective_tools()` call in `create_parashakti_agent()`

**Neo4j Schema Requirements**:
- BimbaResonance nodes MUST have BOTH `EA` and `Episodic` labels (for proper querying)
- `etymology_session_id` property copied from community for session-level queries
- Relationships: `(community)-[:RESONATES_WITH]->(resonance)-[:TARGETS]->(coord)`

**DeepSeek Model**:
- Model: `deepseek:deepseek-chat` (64K context, reasoning-capable)
- Exposes internal reasoning chain in response (transparency)
- Store reasoning chain in `deepseek_reasoning_chain` property

**Re-analysis Workflow**:
- Users can manually re-trigger MEF analysis (UI button)
- `clear_existing_resonances()` removes old resonances before new analysis
- `is_reanalysis` flag in API response indicates prior resonances existed

**Async Execution**:
- MEF analysis triggered via background task (FastAPI BackgroundTasks)
- Returns immediately to user, analysis runs async
- Poll for results (30-second timeout) or refetch communities
