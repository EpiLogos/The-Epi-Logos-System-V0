5. If an invalid coordinate string is provided, the function must return an error

### Context Frame Implementation Details (v2.1)

**Critical Architectural Decision**: Context frames MUST be treated as atomic units during parsing.

**Why This Matters**:
Without context frame recognition, a coordinate like `#1-3-4.0/1/2/3` would be incorrectly parsed as:
```
❌ WRONG: [1, 3, 4, 0, 1, 2, 3]  // Splits context frame into individual numbers
✅ CORRECT: [1, 3, 4, "0/1/2/3"] // Preserves context frame as atomic unit
```

The WRONG parsing would be **completely useless** for Paramasiva's QL mathematical operations, which need to know:
1. Base coordinate components: `[1, 3, 4]`
2. Applied context frame: `"0/1/2/3"` (Dual-Track Parallel Equation)

**Implementation Approach**:
1. Added `CONTEXT_FRAMES` constant array with 6 recognized patterns
2. Updated `parse_coordinate()` to return `tuple[List[int | str], str | None]`
3. Parser checks for context frame suffix after `.` or `-` separator
4. Context frame extracted and preserved as final component
5. Updated `CoordinateComponents` model:
   - `components`: Changed from `List[int]` to `List[int | str]`
   - Added `context_frame: str | None` field

**Testing Coverage**:
- All 6 context frames tested individually
- Both dot (`.`) and hyphen (`-`) separators before context frames
- Mixed coordinates with and without context frames
- Edge cases and error conditions

**Example Outputs**:
```python
# Standard coordinate (no context frame)
decompose_coordinate("#1-2-3")
# → components=[1, 2, 3], context_frame=None

# Coordinate with context frame
decompose_coordinate("#1-3-4.0/1/2/3")
# → components=[1, 3, 4, "0/1/2/3"], context_frame="0/1/2/3"

# All context frames supported
decompose_coordinate("#1.0000")        # → [1, "0000"], context_frame="0000"
decompose_coordinate("#2-1.0/1")       # → [2, 1, "0/1"], context_frame="0/1"
decompose_coordinate("#3.4.0-4.4/5")   # → [3, "4.0-4.4/5"], context_frame="4.0-4.4/5"
```
